# DOD.md — Definition of Done (Benchcraft Web v1.1)

Этот документ отвечает на вопрос: **когда шаг считается завершённым**.

> Цель: уменьшить “тонну правок в конце” и держать качество на каждом маленьком диффе.

---

## A) Минимальный DoD для любого чекпоинта (15–30 минут)

1) **Цель шага достигнута** (как записано в `plans/NOW.md`).
2) **Нет поломок**:
   - приложение не падает на базовом запуске (если применимо),
   - нет очевидных runtime errors.
3) **Гейты качества** (когда кодовая база уже существует):
   - `lint` пройден,
   - `typecheck` пройден.
4) **Контроль изменений**:
   - дифф небольшой и ревьюится человеком,
   - изменения описаны в отчёте (`reports/`).
5) **Зависимости**:
   - новые зависимости — только с явным апрувом пользователя.
6) **UI (если трогали UI)**:
   - нет явных проблем с отступами/выровненностью,
   - состояния (hover/focus/disabled) не забыты,
   - базовая адаптивность не сломана.

---

## B) DoD для “существенного изменения” (feature / refactor)

Дополнительно к A:
1) Есть **ExecPlan** (файл плана) с шагами и критериями.
2) Обновлены/созданы:
   - `docs/UI_SPEC.md` (если это UI‑фича),
   - `docs/STYLE_GUIDE.md` (если меняли палитру/типографику/токены),
   - `docs/ARCHITECTURE.md` (если меняли архитектуру).
3) Тесты:
   - минимум unit/integration на критичный код,
   - e2e — по необходимости (когда фича большая или критичная).
4) Безопасность:
   - секреты не закоммичены,
   - проверены потенциальные инъекции/уязвимости там, где это релевантно.
5) Документация:
   - есть “как проверить руками” чеклист.

---

## C) Что считается “не done”

- “Работает у меня, но не запускалось/не проверялось”
- “Починил тесты, отключив их”
- “Добавил библиотеку без согласования”
- “UI выглядит нормально, но spacing ‘на глаз’ и потом будет боль”
